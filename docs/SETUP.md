## Firebase Setup

### 1. إنشاء مشروع Firebase
1. اذهب إلى [console.firebase.google.com](https://console.firebase.google.com)
2. أنشئ مشروع جديد
3. فعّل **Firestore Database** (Production mode)
4. انسخ إعدادات المشروع

### 2. ضبط firebase-config.js
```js
const firebaseConfig = {
  apiKey:            "AIzaSy...",
  authDomain:        "project-id.firebaseapp.com",
  projectId:         "project-id",
  storageBucket:     "project-id.appspot.com",
  messagingSenderId: "123456789",
  appId:             "1:123456789:web:abcdef"
};
```

### 3. تفعيل Firebase Authentication (مهم — أمان حقيقي)
منذ v1.1.7، التطبيق بيستخدم **Firebase Authentication الحقيقي** بدل التحقق الشكلي القديم. لازم تفعّله يدوياً مرة واحدة بس:

1. من قائمة Firebase Console الجانبية: **Authentication** ← **Get started**
2. من تبويب **Sign-in method**، فعّل مزوّد **Email/Password** فقط (زرار Enable ← Save)
3. مفيش أي حاجة تانية مطلوبة — التطبيق بيحوّل اسم المستخدم لإيميل وهمي (`username@suqia.local`) من تحت تلقائياً، والمستخدم نفسه هيفضل يكتب اسم مستخدم وكلمة مرور عادي زي دايماً من غير ما يحس بأي فرق.

> 💡 **ترحيل تلقائي بصمت:** أي مستخدم قديم (اتعمله حساب قبل v1.1.7) هيترحّل تلقائياً لـ Firebase Auth الحقيقي أول مرة يسجّل دخول بنجاح بعد التحديث — من غير ما يحتاج يعمل أي حاجة أو يغيّر كلمة مروره.

### 4. نشر قواعد Firestore الحقيقية
الملف `firestore.rules` في جذر المشروع فيه القواعد الجاهزة. لنشرها:
1. من Firebase Console: **Firestore Database** ← تبويب **Rules**
2. افتح ملف `firestore.rules` من المشروع، انسخ محتواه بالكامل
3. الصقه في محرر القواعد بالكونسول واضغط **Publish**

القواعد دي بقت بتتحقق فعلياً من هوية المستخدم وصلاحيته (مسؤول/محرر/مشاهد) عن طريق `request.auth`، مش مجرد شكل ظاهري كان قبل كده.

> ⚠️ **حدود معروفة (بصراحة كاملة):** بما إن مفيش خادم خلفي (Cloud Functions) في المعمارية دي:
> - المسؤول **مش يقدر** يغيّر كلمة مرور مستخدم تاني مباشرة (Firebase Auth من غير Admin SDK بيسمح بس للمستخدم يغيّر كلمة مروره هو نفسه). كل مستخدم بيغيّر كلمة مروره بنفسه من تبويب "الحساب".
> - حذف مستخدم من الإعدادات بيمنعه من الدخول فعلياً (بيشيل ملفه الشخصي)، لكن حساب Firebase Auth الأساسي بتاعه بيفضل موجود تقنياً — يعني **متقدرش تنشئ مستخدم جديد بنفس اسم المستخدم القديم بعد حذفه** (هيديك خطأ "مُستخدم من قبل"). لو محتاج الإمكانية دي، الحل الصحيح هو إضافة Cloud Function صغيرة بصلاحيات Admin SDK — خطوة مستقبلية اختيارية لو احتجناها فعلاً.
> - **قراءة مجموعة `users` مسموحة حتى من غير تسجيل دخول عمداً** (`allow read: if true`) — مش هفوة. مستخدم قديم لسه ما ترحّلش لـ Firebase Auth الحقيقي، التطبيق محتاج يتحقق من الـ passwordHash القديم بتاعه *قبل* ما يكون عنده جلسة موثّقة أصلاً (وده تحديداً سبب الترحيل). الكتابة (تعديل/حذف) تفضل للمسؤول فقط دايماً بدون أي تنازل. لو كل المستخدمين اترحّلوا خلاص وحبيت تشدّد القاعدة دي أكتر، غيّرها لـ `allow read: if isSignedIn();` في `firestore.rules`.

## تشغيل التطبيق

### خيار 1: Python (بدون تثبيت)
```bash
cd suqia
python -m http.server 7070
# افتح: http://localhost:7070
```

### خيار 2: Node.js serve
```bash
npm install -g serve
serve . -p 7070
```

### خيار 3: VS Code Live Server
- ثبّت امتداد Live Server
- click بالزر الأيمن على `index.html` ← Open with Live Server

## استيراد البيانات

### أول مرة فقط
1. سجّل دخول بحساب المسؤول الأول المتفق عليه (هيترحّل تلقائياً لنظام Firebase Auth الحقيقي أول مرة تدخل بنجاح)
2. اذهب لـ **الإعدادات**

## إدارة المستخدمين

| الإجراء | الخطوات |
|---------|---------|
| إضافة مستخدم | الإعدادات ← + مستخدم جديد |
| تغيير كلمة مرورك الخاصة | الإعدادات ← تبويب الحساب ← 🔒 تغيير كلمة المرور |
| تعديل اسم/صلاحية مستخدم | الإعدادات ← ✏️ تعديل |
| حذف مستخدم | الإعدادات ← ✏️ تعديل ← 🗑 حذف |

> منذ v1.1.5: 3 صلاحيات حقيقية — 👑 مسؤول (كل شيء) / ✏️ محرر (إضافة وتعديل البيانات) / 👁️ مشاهد (عرض فقط). المسؤول فقط يقدر يدير المستخدمين ويشوف سجل النشاط وقاعدة البيانات.

## استكشاف الأخطاء

| المشكلة | السبب | الحل |
|---------|-------|------|
| خطأ في الاتصال بالخادم | Firestore Rules أو Authentication مش مفعّلين صح | راجع قسم 3 و4 فوق |
| لا توجد بيانات | لم يتم الاستيراد | الإعدادات ← ابدأ الاستيراد |
| تعذّر تحميل البيانات | timeout 6 ثواني | تحقق من الإنترنت ثم أعد المحاولة (أو هيظهر آخر نسخة محفوظة أوفلاين تلقائياً) |
| نسيت كلمة المرور | — | لسه محتاج حل — كل مستخدم بيغيّر كلمة مروره بنفسه بس وهو مسجّل دخول. لو نسيها فعلاً، محتاج Cloud Function بصلاحيات Admin (خطوة مستقبلية اختيارية) |
| "اسم المستخدم موجود من قبل" عند إعادة إنشاء مستخدم محذوف | حساب Firebase Auth الأساسي لسه موجود تقنياً حتى بعد حذف الملف الشخصي | استخدم اسم مستخدم مختلف، أو راجع قسم 4 فوق |

## بنية Firestore

```
Firestore/
├── users/
│   └── {username}/      ← name, role ('admin'|'editor'|'viewer'), passwordHash (قديم/ترحيل فقط — الباسورد الحقيقي بقى في Firebase Auth)
├── activityLog/
│   └── {autoId}/        ← username, name, action, ts (تسجيل دخول/خروج)
├── specs/
│   └── {id}/            ← name, eng, sections[], ...
├── bridges/
│   └── {id}/            ← name, canalName, type, load, ...
├── linedCanals/
│   └── {id}/            ← name, district, liningType, ...
├── wells/
│   └── {id}/            ← name, district, depth, ...
└── drains/
    └── {id}/            ← name, eng, canalName, bank, length, zomam, ... (نموذج مبسّط، بدون مناوبة أو تبطين — منذ v1.4.0)
```

## ملاحظة للمطورين: الاختبار من غير وصول حقيقي لـ Firebase

في بيئات تطوير مقفولة الشبكة (مفيش وصول لـ `gstatic.com` أو `firestore.googleapis.com`)، منهجية الاختبار المتّبعة:

1. انسخ `js/firebase-config.js` كامل، واستبدل **فقط** كتلة الـ `import` الخاصة بـ Firebase SDK (أول ~40 سطر: `initializeApp`, `getFirestore`, `collection`, `getDocs`, `addDoc`, إلخ) بدوال وهمية بسيطة (in-memory، أو محفوظة في `localStorage` عشان تعيش بعد `location.reload()`).
2. **كل الكود اللي تحت الاستيراد يفضل زي ما هو من غير أي تعديل** — بيضمن إن الاختبار بيغطي المنطق الحقيقي (الكاش، الصلاحيات، سجل النشاط...) مش نسخة مبسّطة منه.
3. استخدم `sessionStorage.setItem('suqia_user', ...)` لحقن جلسة مستخدم وهمية قبل فتح أي صفحة، بدل المرور بصفحة تسجيل الدخول فعلياً.
4. افتح الصفحات بمتصفح حقيقي (Playwright/Puppeteer + Chromium headless) وتفاعل معاها فعلياً (كليك، ملء نماذج، فحص DOM) بدل افتراض إن الكود هيشتغل.
5. ⚠️ درس مهم: أي `async` function بتتنادى من غير `await` قبل `location.href=...` أو `location.reload()` ممكن تتلغي كتابتها للشبكة قبل ما تخلص — اتأكد دايماً من الـ `await` قبل أي انتقال.
