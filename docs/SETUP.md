## إعداد Firebase

### 1. إنشاء مشروع Firebase
1. الانتقال إلى [console.firebase.google.com](https://console.firebase.google.com)
2. إنشاء مشروع جديد
3. تفعيل **Firestore Database** (بوضع Production mode)
4. نسخ إعدادات المشروع

### 2. ضبط ملف firebase-config.js
```js
const firebaseConfig = {
  apiKey:            "AIzaSy...",
  authDomain:        "project-id.firebaseapp.com",
  projectId:         "project-id",
  storageBucket:     "project-id.appspot.com",
  messagingSenderId: "123456789",
  appId:             "1:123456789:web:abcdef"
};
```

### 3. تفعيل Firebase Authentication (خطوة مهمة — أمان حقيقي)
منذ الإصدار 1.1.7، يعتمد التطبيق على **Firebase Authentication الحقيقي** بدلاً من التحقق الشكلي القديم. يجب تفعيله يدوياً مرة واحدة فقط:

1. من القائمة الجانبية في Firebase Console: **Authentication** ← **Get started**
2. من تبويب **Sign-in method**، يُفعَّل مزوِّد **Email/Password** فقط (زر Enable ← Save)
3. لا حاجة إلى أي خطوة إضافية — يحوِّل التطبيق اسم المستخدم تلقائياً إلى بريد إلكتروني صوري (`username@suqia.local`) من الخلف، بينما يستمر المستخدم في إدخال اسم مستخدم وكلمة مرور عاديين كالمعتاد، من دون أن يلحظ أي فرق.

> 💡 **ترحيل تلقائي وصامت:** أي حساب قديم (أُنشئ قبل الإصدار 1.1.7) يُرحَّل تلقائياً إلى Firebase Auth الحقيقي عند أول تسجيل دخول ناجح بعد التحديث، من دون الحاجة إلى أي إجراء إضافي أو تغيير في كلمة المرور.

### 4. نشر قواعد Firestore الفعلية
يحتوي الملف `firestore.rules` في جذر المشروع على القواعد الجاهزة للنشر:
1. من Firebase Console: **Firestore Database** ← تبويب **Rules**
2. يُفتح ملف `firestore.rules` من المشروع، ويُنسخ محتواه بالكامل
3. يُلصَق في محرِّر القواعد بلوحة التحكم، ثم يُضغط زر **Publish**

تتحقق هذه القواعد فعلياً من هوية المستخدم وصلاحيته (مسؤول/محرر/مشاهد) عبر `request.auth`، وليست مجرد شكل ظاهري كما كان الحال سابقاً.

> ⚠️ **قيود معروفة (بمنتهى الشفافية):** نظراً لعدم وجود خادم خلفي (Cloud Functions) في هذه المعمارية:
> - **لا يستطيع المسؤول** تغيير كلمة مرور مستخدم آخر مباشرة (لا تسمح خدمة Firebase Auth، من دون استخدام Admin SDK، إلا للمستخدم نفسه بتغيير كلمة مروره). يقوم كل مستخدم بتغيير كلمة مروره بنفسه من تبويب "الحساب".
> - حذف مستخدم من الإعدادات يمنعه فعلياً من تسجيل الدخول (بحذف ملفه الشخصي)، غير أن حساب Firebase Auth الأساسي الخاص به يبقى موجوداً تقنياً؛ ولذلك **لا يمكن إنشاء مستخدم جديد بالاسم القديم نفسه بعد حذفه** (ستظهر رسالة خطأ "مُستخدَم من قبل"). الحل الصحيح لهذه الإمكانية هو إضافة دالة سحابية صغيرة (Cloud Function) بصلاحيات Admin SDK — وهي خطوة مستقبلية اختيارية عند الحاجة الفعلية إليها.
> - **صلاحية قراءة مجموعة `users` أصبحت (منذ الإصدار 1.4.1) مقتصرة على المستخدمين المسجَّلين فقط** (`allow read: if isSignedIn();`)، بعد أن كانت في الإصدارات السابقة مفتوحة للجميع عمداً (`allow read: if true`) لتمكين التحقق من الحسابات القديمة قبل تسجيل الدخول. كان لهذا القرار القديم أثر جانبي خطير: تعريض حقل `passwordHash` (القيمة المُجزَّأة بخوارزمية SHA-256 مع قيمة تمليح ثابتة) لأي شخص على الإنترنت، وهو ما كان يتيح هجوم القوة الغاشمة (Brute-force) دون اتصال على أي حساب لم يُرحَّل بعد. أثر ذلك التشديد: أي حساب قديم لم يُرحَّل بعد إلى Firebase Auth الحقيقي سيتلقى الآن خطأ صلاحيات واضحاً عند محاولة تسجيل الدخول، مصحوباً برسالة تطلب التواصل مع المسؤول لإتمام الترحيل يدوياً. تُحذف قيمة `passwordHash` تلقائياً من كل مستند فور اكتمال ترحيله، وتبقى صلاحية الكتابة (تعديل المستخدمين أو حذفهم) مقتصرة على المسؤول دوماً، من دون أي استثناء.

## تشغيل التطبيق

### الخيار الأول: خادم Python (من دون تثبيت)
```bash
cd suqia
python -m http.server 7070
# يُفتح على المتصفح: http://localhost:7070
```

### الخيار الثاني: أداة serve الخاصة بـ Node.js
```bash
npm install -g serve
serve . -p 7070
```

### الخيار الثالث: امتداد Live Server في VS Code
- يُثبَّت امتداد Live Server
- نقرة بزر الفأرة الأيمن على `index.html` ← Open with Live Server

## إدارة المستخدمين

| الإجراء | الخطوات |
|---------|---------|
| إضافة مستخدم | الإعدادات ← + مستخدم جديد |
| تغيير كلمة المرور الخاصة بالمستخدم نفسه | الإعدادات ← تبويب الحساب ← 🔒 تغيير كلمة المرور |
| تعديل اسم مستخدم أو صلاحيته | الإعدادات ← ✏️ تعديل |
| حذف مستخدم | الإعدادات ← ✏️ تعديل ← 🗑 حذف |

> منذ الإصدار 1.1.5: توجد ثلاث صلاحيات حقيقية — 👑 مسؤول (كل الصلاحيات) / ✏️ محرر (إضافة وتعديل البيانات) / 👁️ مشاهد (عرض فقط). يملك المسؤول وحده صلاحية إدارة المستخدمين، والاطلاع على سجل النشاط وقاعدة البيانات.

## استكشاف الأخطاء وحلولها

| المشكلة | السبب | الحل |
|---------|-------|------|
| خطأ في الاتصال بالخادم | لم تُفعَّل قواعد Firestore أو خدمة Authentication بالشكل الصحيح | تُراجَع الخطوتان 3 و4 أعلاه |
| لا توجد بيانات | لم يتم الاستيراد بعد | الإعدادات ← ابدأ الاستيراد |
| تعذّر تحميل البيانات | انتهت مهلة الاتصال (10 ثوانٍ) | يُتحقَّق من الاتصال بالإنترنت ثم تُعاد المحاولة (أو تظهر تلقائياً آخر نسخة محفوظة محلياً من دون اتصال) |
| نسيان كلمة المرور | — | لا يزال بحاجة إلى حل شامل — يستطيع كل مستخدم حالياً تغيير كلمة مروره بنفسه فقط أثناء تسجيل دخوله. أما في حال نسيانها فعلياً، فالأمر يتطلب دالة سحابية (Cloud Function) بصلاحيات Admin (خطوة مستقبلية اختيارية) |
| ظهور رسالة "اسم المستخدم موجود من قبل" عند إعادة إنشاء مستخدم محذوف | لا يزال حساب Firebase Auth الأساسي موجوداً تقنياً حتى بعد حذف الملف الشخصي | يُستخدَم اسم مستخدم مختلف، أو تُراجَع الفقرة الرابعة أعلاه |

## بنية Firestore

```
Firestore/
├── users/
│   └── {username}/      ← name, role ('admin'|'editor'|'viewer'), passwordHash (للحسابات القديمة قيد الترحيل فقط — كلمة المرور الفعلية أصبحت في Firebase Auth)
├── activityLog/
│   └── {autoId}/        ← username, name, action, ts (تسجيل دخول أو خروج)
├── specs/
│   └── {id}/            ← name, eng, sections[], ...
├── bridges/
│   └── {id}/            ← name, canalName, type, load, ...
├── linedCanals/
│   └── {id}/            ← name, district, liningType, ...
├── wells/
│   └── {id}/            ← name, district, depth, ...
└── drains/
    └── {id}/            ← name, eng, canalName, bank, length, zomam, ... (نموذج مبسّط، من دون مناوبة أو تبطين — منذ الإصدار 1.4.0)
```

## ملاحظة للمطورين: الاختبار من دون وصول فعلي إلى Firebase

في بيئات التطوير المقفولة عن الشبكة (حيث لا يوجد وصول إلى `gstatic.com` أو `firestore.googleapis.com`)، تُتَّبع منهجية الاختبار التالية:

1. تُنسخ نسخة كاملة من الملف `js/firebase-config.js`، وتُستبدَل **فقط** كتلة الاستيراد (`import`) الخاصة بحزمة Firebase SDK (نحو الأربعين سطراً الأولى: `initializeApp`, `getFirestore`, `collection`, `getDocs`, `addDoc`، وغيرها) بدوال وهمية بسيطة (تعمل في الذاكرة، أو تُحفظ في `localStorage` كي تبقى قائمة بعد تنفيذ `location.reload()`).
2. **يبقى كل الكود الواقع بعد كتلة الاستيراد كما هو من دون أي تعديل** — لضمان أن يغطي الاختبار المنطق الحقيقي (التخزين المؤقت، والصلاحيات، وسجل النشاط...) لا نسخة مبسَّطة منه.
3. يُستخدَم الأمر `sessionStorage.setItem('suqia_user', ...)` لحقن جلسة مستخدم وهمية قبل فتح أي صفحة، بدلاً من المرور فعلياً بصفحة تسجيل الدخول.
4. تُفتح الصفحات بمتصفح حقيقي (Playwright أو Puppeteer مع Chromium بوضع headless)، ويُتفاعَل معها فعلياً (نقر، وملء نماذج، وفحص شجرة DOM) بدلاً من افتراض أن الكود سيعمل كما هو متوقَّع.
5. ⚠️ ملاحظة مهمة: أي دالة `async` تُستدعى من دون `await` قبل تنفيذ `location.href=...` أو `location.reload()` قد يُلغى تنفيذ عملية الكتابة إلى الشبكة الخاصة بها قبل اكتمالها؛ لذا يجب التأكد دائماً من استخدام `await` قبل أي عملية انتقال بين الصفحات.
