## نظرة عامة

**سُقيا** منصة ويب متكاملة لإدارة وتوثيق الأصول الهندسية لري محافظة الجيزة، مبنية بالكامل باستخدام Vanilla JS (من دون أُطر عمل) وقاعدة بيانات Firebase Firestore.

| الوحدة | الوصف | مجموعة Firestore |
|--------|-------|----------------------|
| 📊 لوحة التحكم | نظرة عامة على الأصول، وبحث موحّد | — |
| 🌊 ترع الإدارة | الأورنيك الهندسي للترع (المقاطع، والمناوبات، والتبطين) | `specs` + `linedCanals` |
| 🌉 الكباري | كباري الترع والمصارف | `bridges` |
| 🚰 المصارف | مصارف الصرف الزراعي — نموذج مبسّط من دون مناوبة أو تبطين | `drains` |
| 💧 الآبار الجوفية | آبار المياه الجوفية وبيانات الطلمبات | `wells` |
| 🗺️ الخريطة | خريطة تفاعلية موحّدة (Leaflet) لجميع الأصول ذات الإحداثيات الجغرافية | — |
| ⚙️ الإعدادات | الحساب، والمستخدمون والصلاحيات، وسجل النشاط، وقاعدة البيانات | `users`, `activityLog` |

## التشغيل السريع

### المتطلبات
- مشروع Firebase مُفعَّل عليه **Firestore وAuthentication** معاً (أصبح تفعيل Authentication خطوة إلزامية منذ الإصدار 1.1.8)
- خادم ويب ثابت (VS Code Live Server، أو Python، أو Nginx، أو Cloudflare Pages)

### الخطوات

1. **استنساخ المشروع**
   ```bash
   git clone <repo-url>
   cd suqia
   ```

2. **ضبط إعدادات Firebase** — في الملف `js/firebase-config.js`:
   ```js
   const FIREBASE_CONFIG = {
     apiKey: "YOUR_KEY",
     authDomain: "YOUR_PROJECT.firebaseapp.com",
     projectId: "YOUR_PROJECT",
     // ...
   };
   ```

3. **تفعيل Firebase Authentication** — من Firebase Console: Authentication ← Sign-in method ← تُفعَّل خاصية **Email/Password**. (التفاصيل الكاملة في `SETUP.md`)

4. **نشر قواعد Firestore** — يُنسخ محتوى الملف `firestore.rules` من جذر المشروع، ويُلصَق في Firebase Console ← Firestore Database ← Rules ← Publish.

5. **تشغيل الخادم محلياً**
   ```bash
   # Python
   python -m http.server 7070
   # أو Node.js
   npx serve .
   ```

6. **تسجيل الدخول لأول مرة** بحساب المسؤول الأول — يُرحَّل تلقائياً إلى نظام Firebase Auth الحقيقي عند أول تسجيل دخول ناجح.

7. **استيراد البيانات** — من الإعدادات ← ابدأ الاستيراد.

## دعم الهاتف المحمول

- شريط تنقل سفلي ثابت (Bottom Navigation) في جميع الصفحات، مع تفعيل تلقائي للعنصر النشط
- زر إضافة عائم (FAB) بدلاً من وضعه داخل شريط العنوان، لتفادي ازدحام الرأسية على الشاشات الضيقة
- البحث الموحّد موجود في لوحة التحكم فقط، تفادياً لازدحام شريط العنوان في بقية الصفحات
- مساحات لمس مريحة (40 بكسل فأكثر) لجميع الأزرار والأيقونات
- جرى فحص كل صفحة فعلياً بمتصفح حقيقي على عرض 375 بكسل، للتأكد من خلوّها من أي تمدد أفقي أو محتوى مبتور

## نظام المصادقة والصلاحيات

- **مصادقة حقيقية عبر Firebase Authentication** — اسم مستخدم وكلمة مرور عاديان، يتحوّلان تلقائياً إلى بريد إلكتروني داخلي وهمي (`username@suqia.local`) من دون أن يلحظ المستخدم أي فرق. اتُّخذ هذا القرار بدلاً من الاعتماد على مصادقة مخصصة، حتى تتمكن قواعد Firestore من التحقق فعلياً من هوية المستخدم عبر `request.auth`.
- تُرحَّل الحسابات القديمة (من قبل الإصدار 1.1.8) تلقائياً وبصمت إلى Firebase Auth عند أول تسجيل دخول ناجح، من دون الحاجة إلى أي إجراء إضافي.
- تُحفظ الجلسات في `sessionStorage` (للواجهة، ولتحديد آخر مستخدم مسجَّل)، إضافة إلى جلسة Firebase Auth حقيقية (للحماية الفعلية في قواعد Firestore).
- ثلاث صلاحيات حقيقية: 👑 مسؤول (كل شيء، بما في ذلك إدارة المستخدمين) / ✏️ محرر (إضافة وتعديل البيانات) / 👁️ مشاهد (عرض فقط، من دون أي أزرار تعديل).
- سجل نشاط: تُسجَّل كل عملية دخول أو خروج في `activityLog`، وتظهر للمسؤول من تبويب الإعدادات.

> ⚠️ **قيود معروفة (نظراً لعدم وجود خادم خلفي — Cloud Functions):**
> - لا يستطيع المسؤول تغيير كلمة مرور مستخدم آخر مباشرة — يقوم كل مستخدم بتغييرها بنفسه من تبويب "الحساب".
> - يمنع حذف المستخدم من تسجيل الدخول فعلياً، غير أن حساب Firebase Auth الخاص به يبقى موجوداً تقنياً؛ ولذلك لا يمكن إنشاء مستخدم جديد بالاسم القديم نفسه بعد حذفه.
> - التفاصيل الكاملة وأسبابها التقنية، والحل المستقبلي المقترح (دالة سحابية اختيارية)، موضَّحة في [`SETUP.md`](SETUP.md).
